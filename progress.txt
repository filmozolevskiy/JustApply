## Iteration 2 — Issue #37: Persist Task Logs across dashboard refresh

**What was implemented:**

The frontend persistence (localStorage session history, active task reconnect on refresh) and server-side `skip` parameter for the SSE log stream were already partially implemented from prior work. This iteration completed and verified the full feature:

**Server-side (`src/web/server.py`):**
- Added `skip: int = 0` query parameter to `GET /api/logs/{task_id}`
- Negative skip values are clamped to 0
- The SSE generator yields `state.logs[skip:]` instead of all logs, so reconnecting clients skip already-received lines

**Tests (`tests/test_log_stream.py`) — 8 new TDD tests:**
- `test_unknown_task_returns_404` — unknown task_id returns 404
- `test_skip_zero_returns_all_logs` — skip=0 yields all pre-existing log lines
- `test_skip_two_omits_first_two_logs` — skip=2 skips first 2 entries
- `test_skip_equal_to_total_yields_no_logs` — skip at log count yields nothing
- `test_negative_skip_is_treated_as_zero` — skip=-5 behaves like skip=0
- `test_completed_task_stream_ends_with_done_event` — done event is sent
- `test_failed_task_stream_ends_with_error_and_done` — error log + done on failure
- `test_skip_one_then_skip_two_covers_all_logs` — simulates two reconnects without overlap

**Frontend (`src/web/dashboard.html`) — already committed in #36, verified:**
- `sessionLogs` array capped at 200 entries, persisted to `localStorage` on every `addLogLine()`
- `restoreSessionLogs()` repopulates the console on page load
- `clearLogs()` wipes both the UI and `localStorage`
- `ACTIVE_SCRAPE_TASK_KEY` / `ACTIVE_ENRICH_TASK_KEY` and matching skip-counter keys persisted
- `restoreActiveScrapeTask()` and `restoreActiveEnrichTask()` reconnect SSE streams with correct `?skip=N` on refresh
- `connectTaskLogStream()` removes task/skip keys from `localStorage` on task completion

**All 204 tests pass.**

## Iteration 4 — Issue #39: Active Contact greeting name substitution

**What was implemented:**

**`src/web/dashboard.html` — greeting substitution helpers:**
- Added `NAME_PLACEHOLDER = '______'` constant
- Added `applyGreetingName(template, firstName)` — regex-replaces the name token in the greeting line (`Hi <name>,` / `Dear <name>,`) with the contact's first name; works whether the token is `______` or a previously substituted name
- Added `normalizeGreeting(template)` — restores `NAME_PLACEHOLDER` in the greeting line before saving, so stored templates always use the placeholder

**`selectActiveContact` update:**
- After loading the audience template via `getActiveTemplate`, extracts the contact's first name (first word of `contact.name`) and calls `applyGreetingName` before setting `textarea.value`

**`openJobDetailsDrawer` update:**
- Applies `applyGreetingName` for the default active contact so the greeting shows the correct first name when the drawer first opens

**`saveOutreachTemplate` update:**
- Calls `normalizeGreeting(template)` before updating in-memory state and before sending the `PUT /api/jobs/{id}/template` request, ensuring the stored template always holds `______`

**Tests (`tests/test_dashboard_outreach_settings.py`) — 9 new TDD tests:**
- `test_name_placeholder_constant_defined` — `NAME_PLACEHOLDER` constant exists in the script
- `test_apply_greeting_name_function_defined` — `applyGreetingName` function is defined
- `test_normalize_greeting_function_defined` — `normalizeGreeting` function is defined
- `test_select_active_contact_applies_greeting_substitution` — `selectActiveContact` calls `applyGreetingName`
- `test_save_outreach_template_normalizes_greeting` — `saveOutreachTemplate` calls `normalizeGreeting`
- `test_open_job_drawer_applies_greeting_for_default_contact` — `openJobDetailsDrawer` calls `applyGreetingName`
- `test_apply_greeting_name_replaces_placeholder_in_hi_greeting` — function body uses `replace`
